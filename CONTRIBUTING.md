# Contributing to Oh My Cassette

Thanks for helping! A few conventions keep releases automatic and reviews fast.

## Conventional commits (required)

Releases, version bumps, and CHANGELOG entries are generated by
[release-please](https://github.com/googleapis/release-please) from commit
messages, so commits on `main` must follow
[Conventional Commits](https://www.conventionalcommits.org/):

- `feat: …` — user-visible feature (minor version bump)
- `fix: …` — bug fix (patch bump)
- `feat!: …` or a `BREAKING CHANGE:` footer — breaking change (major bump)
- `docs: …`, `ci: …`, `chore: …`, `test: …`, `refactor: …` — no release impact

PRs are squash-merged with the **PR title as the commit message**, so the PR
title itself must be a conventional commit line.

## Development setup

```bash
uv venv .venv
uv pip install --python .venv/bin/python -r requirements-web.txt pytest
.venv/bin/python -m playwright install chromium
```

Run the checks CI runs:

```bash
python3 -m compileall -q .
.venv/bin/python -m pytest -q -rs
```

Real gateway/Cassette E2E tests are opt-in (`RUN_CASSETTE_E2E=1`); see the
README Development section for details.

## Pull requests

- CI must be green, including the `install-smoke` job, which exercises the
  official `hermes plugins install` flow against your branch.
- If you add or rename a tool or hook, update `plugin.yaml` —
  `tests/test_plugin_manifest.py` enforces that the manifest matches what
  `register()` actually registers.
- Keep `README.md` and `README.zh-cn.md` in sync when touching docs.
- Never commit credentials or runtime state; see [SECURITY.md](./SECURITY.md).
